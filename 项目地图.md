# 项目地图

这个文件是给人看的中文导航页。

它不是代码，也不是数据库，而是用来快速回答三个问题：

- 这个 Personal Brain 现在做到哪一步了
- 每个重要文件负责什么
- 下一步应该先改哪里

## 这个项目是什么

Personal Brain 不是普通记事本，也不是关键词搜索工具。

目标是：

```text
你随意输入想法
-> AI 保存原话
-> AI 拆解、润色、结构化为长期记忆
-> 生成语义向量，方便以后 RAG 检索
-> 回答问题时必须拿记忆证据说话
```

核心原则：

- `raw_messages` 永远保存原始输入
- `memories` 保存 AI 改写后的 atomic memory
- `topics` / `entities` 由 AI 动态识别，不写死分类
- 检索优先走 embedding/RAG，不把关键词匹配当主方案
- 敏感信息进加密 vault，不进入 AI、Router、embedding

## 重要文件说明

### `brain.py`

命令行入口。

常用命令：

```powershell
python brain.py init-db
python brain.py ingest "一条想法"
python brain.py embed-memories
python brain.py recall "想问的问题"
python brain.py ask "我之前怎么想的？"
python brain.py memory-list
python brain.py memory-show 1
```

### `personal_brain/schema.py`

数据库地基。

它定义：

- 原始消息表
- AI 抽取审计表
- 长期记忆表
- embedding 表
- topics / entities 表
- 加密 vault 表

### `personal_brain/extractor.py`

记忆提取器。

它负责：

```text
普通输入
-> 调用 AI
-> 判断是否值得长期记忆
-> 拆成 atomic memories
-> 生成 topic/entity/importance/confidence
-> 写入 SQLite
```

### `personal_brain/semantic.py`

语义检索层。

它负责：

```text
memory
-> embedding vector
-> query embedding
-> cosine similarity
-> 返回候选记忆和原始证据
```

### `personal_brain/answer.py`

基于证据回答问题。

它负责：

```text
问题
-> semantic recall
-> AI rerank 证据
-> 只基于证据回答
-> 标注 memory_id / raw_message_id
```

### `personal_brain/router.py`

Memory Router。

它负责生成轻量索引：

- `brain_index.json`
- `memory/topics.json`
- `memory/memory_manifest.json`

Router 的作用是给 Codex 或其他 AI 指路，避免每次全库读取。

它不是 RAG 本身。

### `personal_brain/vault.py`

敏感信息加密仓库。

密码、API key、token 这类内容应该走：

```powershell
python brain.py secure-add
python brain.py secure-list
python brain.py secure-get
```

不要把敏感信息发到飞书记忆入口里。

### `scripts/feishu_bridge.py`

飞书机器人桥接。

它只做入口：

```text
飞书消息
-> PersonalBrain.ingest(...) 或 PersonalBrain.ask(...)
-> 回复飞书
```

真正的记忆、检索、回答逻辑都在 `personal_brain` 核心层。

### `scripts/wxauto_bridge.py`

Windows 微信桌面桥接雏形。

它和飞书一样，只应该做消息收发，不应该拥有大脑逻辑。

## 数据流

记忆写入：

```text
Feishu / WeChat / CLI
-> raw_messages 保存原文
-> AI extraction run 保存模型输出审计
-> memories 保存 AI 改写后的 atomic memory
-> topics/entities 动态关联
-> embedding_model 启用时生成向量
-> build-router 更新轻量索引
```

记忆查询：

```text
用户问题
-> query embedding
-> semantic recall 找候选 memories
-> AI rerank 证据
-> AI 只基于证据回答
```

Codex 长对话读取：

```text
brain_index.json
-> memory/topics.json
-> memory/memory_manifest.json
-> 必要时读取 SQLite 原始证据
```

## 暂时不做

- 不做知识图谱可视化
- 不做 Neo4j
- 不做 GraphRAG
- 不做复杂前端
- 不做多数据库
- 不用固定分类文件夹代替 AI 判断
- 不用关键词搜索冒充 RAG

## 下一步建议

优先顺序：

1. 修好第一版 smoke test，保证 `ingest -> embed -> recall -> ask` 稳定。
2. 把飞书机器人跑通，作为主要输入入口。
3. 写每周 Markdown 整理命令，让 Codex 读取 Router 和 memories 后输出 topics / insights。
4. 再考虑更强的长期记忆合并、冲突检测和重要性更新。
